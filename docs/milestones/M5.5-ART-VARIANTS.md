# M5.5 — Art Variants in Gameplay

> Persist per-card art variant selections from the deck builder through to game rendering. Players see their chosen parallel/alternate art on all cards during gameplay and in deck preview modals.

---

## Goal

Close the gap between deck builder art selection (which exists today but isn't persisted for non-leader cards) and in-game card rendering. A player who selects a Secret Rare parallel in their deck builder should see that art on the board during a game and in all deck preview modals.

Fast follows extend this to DON card art customization and card sleeve selection.

---

### Critical Data Flow

```
Deck Builder → Save API → Database → Lobby Join → toCardData() → Worker Init → cardDb → BoardCard
```

Variant URLs are overlaid onto `cardData.imageUrl` at game init time. No changes needed to `CardData` type, `GameState`, worker engine, or `BoardCard` — the `imageUrl` field already flows through the entire rendering pipeline.

## Unknowns & Design Gaps

These must be resolved before implementation begins.

### 1. Per-Card vs Per-Card-ID Variant Rendering (Blocking Phase 2)

`cardDb` is a `Map<cardId, CardData>`. All copies of the same card share one `CardData` entry with one `imageUrl`. If a player has 4x OP01-025 and selects a parallel art for 2 of them, there's no way to show different art per copy.

**Options:**
- **(A) One variant per card ID per deck** — all copies of a card use the same art. Simpler, covers 99% of use cases.
- **(B) Per-instance art** — `CardInstance` gets an `imageUrl` override field. More complex, changes game state shape.

**Recommendation:** Option A. Players rarely want mixed art for the same card. Revisit if demand arises.

### 2. DON Card Data Source (Blocking Phase 4)

DON cards currently have zero database representation. Questions:
- Are DON cards in the vegapull data source? Do they have variant IDs?
- If not, how do we model DON variants? Manual curation table? Simple URL list?
- How many DON art variants exist across official OPTCG sets?

### 3. Card Sleeve Asset Strategy (Blocking Phase 5)

- Are we using official OPTCG sleeve designs, custom designs, or both?
- Where do sleeve assets live — `/public/images/` or R2 CDN?
- Are sleeves per-deck or per-user (account-level default)?
- Is there a marketplace/unlock concept, or all available to everyone?

### 4. Art Variant URL Validation (Security)

Current `leaderArtUrl` stores a raw URL string. If we continue this pattern for `DeckCard.selectedArtUrl`, users could theoretically inject arbitrary image URLs via API.

**Options:**
- **(A) Store `artVariantId` (FK to ArtVariant)** — secure, validated, but requires migration of existing `leaderArtUrl` field
- **(B) Validate URL matches a known ArtVariant.imageUrl** — server-side check on save
- **(C) Trust the client** — simpler, but opens door to arbitrary images

**Recommendation:** Option A long-term, Option B as pragmatic first step.

### 5. Export/Import Compatibility

Current deck export format: `4x OP01-001` — no variant encoding.
- Should export/import preserve art variant selections?
- If so, what format? E.g., `4x OP01-001 [OP01-001_p1]`?
- Can be deferred to a follow-up, but format decision should be made upfront.

### 6. Opponent Visibility

Assumed behavior (standard for TCGs):
- Opponents see your selected art on face-up cards (character, leader, stage) — **yes**
- Opponents see your sleeve on face-down cards (hand, deck, life) — **yes**
- Spectators (M6) see variants — **yes**, no extra work needed if imageUrl is in cardDb

Confirm these assumptions before implementation.

## Risks

| Risk | Mitigation |
|------|------------|
| `cardDb` keyed by cardId can't support per-copy variant art | Phase 1 design decision: one variant per card ID per deck (Option A). Revisit only if demanded. |
| Arbitrary image URL injection via `selectedArtUrl` | Server-side validation against ArtVariant table on save |
| DON card art not in data pipeline | Manual curation fallback — DON variants are a small, bounded set |
| Sleeve assets need design work | Start with a small curated set (5–10); expand later |
| Migration adds nullable columns | Additive migration, no backfill, zero downtime |
