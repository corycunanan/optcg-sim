# M4.5 — Schema Encoding QA

> Validate, fix, and harden the 2,500+ card effect schemas encoded in M4. Ensure every schema is correct, every deferred card is accounted for, and the engine is ready for integration testing.

---

## Context

M4 delivered a complete first pass: every released OPTCG card has an effect schema encoded across 50 schema files. This was done at speed by encoding agents, which means systematic errors are expected — wrong enum values, missing back-references, duration mismatches, action types used without engine handlers, and cards that should have been deferred but weren't.

M4.5 is the QA gate between "all cards encoded" and "all cards encoded correctly." No new engine features are added here (that's Phase 3). The goal is to build confidence in the schema corpus before using it for gameplay.

### Guiding Principle: Err on the Side of Caution

**A wrong schema is worse than a missing one.** A deferred card falls back to `MANUAL_EFFECT` and is visibly incomplete. A wrong schema silently produces incorrect gameplay — the wrong targets get selected, the wrong conditions gate an effect, or an action resolves with the wrong semantics. Players won't know the card is broken; they'll just lose games to bugs.

During the first encoding pass, there were cases where an encoding agent didn't know how to handle a pattern and **substituted a more familiar one instead of deferring**. For example, encoding a reveal-conditional as an unconditional draw, or mapping a hand-zone cost reduction to a field modifier. These "low-confidence encodings" are the most dangerous class of error because they pass structural validation — the schema is well-formed, it just doesn't match the card.

**The QA stance is aggressive:** any effect that doesn't cleanly map to an established encoding pattern should be flagged for manual review, not assumed correct. When in doubt, defer. The cost of a false positive (re-reviewing a correct schema) is minutes; the cost of a false negative (shipping a wrong schema) is broken gameplay that's hard to trace back to the encoding.

### What M4.5 Is NOT

- **New engine features** — deferred pattern support (REVEAL_CONDITIONAL, HAND_ZONE_MODIFIER, etc.) is Phase 3
- **Integration testing** — full gameplay tests are Phase 4
- **Card text generation** — no LLM-assisted parsing (that's M5)

## Phase 1: Schema Validation Sweep

Objective: find and fix encoding errors across all 50 schema files.

### 1.1 Run the Semantic Linter

Run `lint-schemas.sh` against the full corpus and triage output.

The linter already checks 8 rule categories:
- **F** (Enum validation): F1–F8 — trigger types, event types, cost types, condition types, duration types, target types, keywords, prohibition types
- **A** (Cross-field consistency): A1–A7 — replacement/permanent block integrity, trigger/action presence, `result_ref`/`target_ref` matching, choice branch counts
- **B** (Duration correctness): B1–B3 — instantaneous actions with duration, duration-required actions missing duration, `THIS_BATTLE` scope
- **C** (Filter semantics): C1–C4 — cost/power filter conflicts, `exclude_self` applicability
- **D** (Trigger/action mismatches): D1–D5 — event card trigger types, KO target presence, SET_REST targeting
- **E** (Cost vs action confusion): E1–E3 — TRASH_FROM_HAND placement, REST_SELF placement, DON cost type confusion

**Deliverables:**
- [ ] Linter runs clean on all 50 schema files (0 errors, warnings triaged)
- [ ] Fix log documenting every error category and how it was resolved

### 1.2 Build Engine Coverage Check

Cross-reference schemas against actual engine handler support. The type system defines **89 action types** but only **41 have handlers** in `effect-resolver/resolver.ts`. Any schema using an unhandled action type should either be deferred or flagged for engine implementation.

**Unhandled action types currently in the type system:**

| Category | Unhandled Types |
|----------|----------------|
| Power/stats | `SET_BASE_POWER`, `SET_POWER_TO_ZERO`, `SWAP_BASE_POWER`, `COPY_POWER`, `SET_COST` |
| Keywords | `GRANT_COUNTER` |
| DON | `REST_DON`, `REDISTRIBUTE_DON`, `GIVE_OPPONENT_DON_TO_OPPONENT`, `DISTRIBUTE_DON`, `RETURN_ATTACHED_DON_TO_COST` |
| Life | `ADD_TO_LIFE` (forward-compat alias) |
| Battle | `REDIRECT_ATTACK`, `DEAL_DAMAGE`, `SELF_TAKE_DAMAGE` |
| Meta/flow | `CHOOSE_VALUE`, `WIN_GAME`, `EXTRA_TURN`, `REMOVE_PROHIBITION` |
| Effects | `ACTIVATE_EVENT_FROM_HAND`, `ACTIVATE_EVENT_FROM_TRASH`, `NEGATE_TRIGGER_TYPE` |
| Search | `SEARCH_TRASH_THE_REST` |

**Deliverables:**
- [ ] New linter rule (F9 or similar) that flags schemas using action types not in `ACTION_HANDLERS`
- [ ] Report: which cards use unhandled action types and should be deferred or need engine work
- [ ] Cards misidentified as encodable → moved to `DEFERRED-CARD-EFFECTS.md`

### 1.3 Detect Low-Confidence Encodings

The most dangerous error class: schemas that are structurally valid but semantically wrong. An encoding agent that didn't know how to handle a pattern may have substituted a more familiar one instead of deferring. These pass the linter but produce incorrect gameplay.

**Detection strategies:**

- **Schema-vs-card-text comparison**: For each schema, compare the encoded actions against the original card text in `docs/cards/`. Flag schemas where the card text contains patterns known to be complex (reveal-conditional, hand-zone modifiers, "choose a cost", "the next time you play") but the schema uses simpler actions (unconditional DRAW, field-scoped MODIFY_COST, etc.)
- **Complexity mismatch**: Card text with multiple conditional branches encoded as a single linear action chain
- **Missing conditions**: Card text says "if X" but the schema has no condition block or the condition doesn't match the text
- **Scope mismatches**: Card text says "in your hand" but schema targets field; card text says "base power" but schema uses MODIFY_POWER instead of SET_BASE_POWER
- **Duration mismatches**: Card text says "until the end of your opponent's next turn" but schema says END_OF_TURN
- **Keyword text not reflected**: Card text grants a keyword (Rush, Blocker) but schema uses MODIFY_POWER or similar
- **Deferred-pattern keywords in non-deferred schemas**: Scan all non-deferred schemas for card text containing known deferred-pattern phrases ("reveal", "choose a cost", "in your hand...cost", "the next time you play"). Any match is a high-priority review candidate.

**Deliverables:**
- [ ] Automated scan for deferred-pattern keywords in non-deferred card text
- [ ] Manual review queue of flagged cards with suspected low-confidence encodings
- [ ] Each flagged card either confirmed correct, fixed, or deferred

### 1.4 Extend Linter for First-Pass Patterns

Common agent encoding mistakes the current linter doesn't catch. Add rules as errors surface:

- Action types that exist in the type system but have no handler (covered by 1.2)
- `target_ref` pointing to an `id` that doesn't exist in the action chain
- Costs that reference card zones inconsistent with `card_type` (e.g., `TRASH_SELF` on a Leader)
- Duplicate `card_id` entries across schema files (same card encoded twice)
- `card_type` mismatches vs database (Leader encoded as Character, etc.)
- `once_per_turn: true` on effects that aren't activate effects

**Deliverables:**
- [ ] At least 3 new linter rules added based on patterns found in 1.1
- [ ] Second linter pass runs cleaner than first

### 2.1 Audit Existing Deferrals

Cross-check every card in `DEFERRED-CARD-EFFECTS.md` (currently 38 cards across 7 blocker categories) against its schema file:

- Confirm the card's non-deferred effects are correctly encoded
- Confirm the deferred effect is genuinely unsupported (not just encoded wrong)
- Remove any cards that were deferred but can actually be encoded with existing action types

### 2.2 Identify New Deferrals

From Phase 1 results, cards flagged by the engine coverage check that use unhandled action types need to be triaged:

- **Defer:** Effect genuinely requires an unimplemented action type → add to `DEFERRED-CARD-EFFECTS.md`

## Phase 3: Engine Feature Implementation

Objective: implement handlers for deferred patterns in priority order, unlocking cards for encoding.

### 3.1 Priority: REVEAL_CONDITIONAL (unlocks 19 cards)

The highest-value blocker. Pattern: reveal card(s) from deck/life, branch based on revealed card properties.

**Required engine work:**
- `REVEAL` action already has a handler — verify it returns revealed card data to the action chain
- Add inline condition evaluation against revealed card properties (trait, cost, name, card type)
- Wire `PLAY_CARD` to target a revealed card (not just hand/deck)
- Handle "then place at bottom of deck" cleanup for non-played reveals

**Cards unlocked:** OP01-060, OP07-048, OP08-049, OP11-066/071/073/074/079/081, OP12-058, OP14-044, OP15-065, ST13-007/010/014, ST17-001, ST22-003/006/007/012/016

- [ ] REVEAL_CONDITIONAL pipeline implemented and tested
- [ ] All 19 deferred cards encoded
- [ ] Linter updated if new patterns need validation

### 3.2 HAND_ZONE_MODIFIER (unlocks 7 cards)

Pattern: permanent cost/power modifiers applied to cards while in hand, conditioned on board state.

**Required engine work:**

**Cards unlocked:** OP01-067, OP07-064, OP14-053, OP15-013, OP15-102, ST23-001/002, ST26-001

**Deliverables:**

## Phase 4: Integration Testing

Objective: verify that validated schemas resolve correctly at runtime through the full pipeline.

### 4.1 Scenario Selection

Select representative cards from each major effect pattern for end-to-end testing:

| Pattern | Example Card | What to Verify |
|---------|-------------|----------------|
| On Play → Draw | ST01-004 Nami | Trigger fires, draw resolves, event logged |
| On Play → KO | OP01-006 Zoro | Target prompt, KO execution, DON return |
| Activate: Main | OP01-025 Luffy | Cost payment, once-per-turn tracking, power modifier with duration |
| Counter step | ST01-015 | Counter event, power buff, THIS_BATTLE expiry |
| Replacement | ST22-012 Marco | K.O. intercepted, replacement applied once |
| Prohibition | Cards with CANNOT_ATTACK | Attack blocked, prohibition expiry |
| PLAYER_CHOICE | Cards with "choose one" | Choice prompt, branch execution |
| SEARCH_DECK | Searcher cards | Deck search prompt, card selection, shuffle |
| Permanent modifier | Leader aura buffs | Modifier persists, recalculates on zone change |
| Scheduled action | End-of-turn effects | Scheduled action fires at correct phase |

### 4.2 Test Harness

Build on the existing test infrastructure (`workers/game/src/__tests__/`):

- **Unit tests**: One test per scenario — build minimal `GameState`, run pipeline, assert final state
- **Round-trip tests**: Full action → pipeline → event → trigger → effect → state cycle
- **Edge cases**: Empty targets, insufficient costs, once-per-turn rejection, nested triggers

**Deliverables:**
- [ ] At least 20 integration tests covering the pattern table above
- [ ] All tests pass with current schema corpus
- [ ] CI runs schema linter + integration tests

### 4.3 Regression Safety

After integration tests pass:
- [ ] Linter runs as part of CI (fail on errors, allow warnings)

## Acceptance Criteria

M4.5 is complete when:

1. `lint-schemas.sh` runs clean (0 errors) on all 50 schema files
2. Engine coverage check confirms no schemas use unhandled action types (except documented deferrals)
3. `DEFERRED-CARD-EFFECTS.md` is accurate and up-to-date
4. All deferred blocker categories from Phase 3 are implemented
5. All previously-deferred cards are encoded
6. At least 20 integration tests pass covering major effect patterns
7. Linter + tests run in CI
