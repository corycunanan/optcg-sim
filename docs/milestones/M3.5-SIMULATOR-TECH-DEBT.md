# M3.5 — Simulator Tech Debt

> Structural cleanup and correctness hardening of the M3 game engine and client before M4 adds effect automation. Split oversized files, eliminate type duplication, add test coverage, decompose the game board UI, migrate inline styles to the design system, fix rules-compliance bugs, and fill data model gaps that M4 depends on.

---

## Why This Phase Exists

M3 delivered a functional OPTCG simulator — two players can play a full game with rules-enforced turns, combat, and keyword automation. The engine architecture (7-step pipeline, immutable state, event bus) is sound. But the implementation accumulated structural debt and rules-compliance gaps that will compound when M4 adds the effect engine:

- **`execute.ts` was a 704-line monolith** — phase transitions, battle logic, card play, DON attachment, concede, and manual effects all lived in one file.
- **Game types were manually duplicated in 3 places** — `workers/game/src/types.ts`, `src/hooks/use-game-ws.ts`, and `src/components/game/game-board.tsx`.
- **No test coverage** — the engine is pure functions but had zero tests.
- **`game-board.tsx` was 1164 lines** — the entire game UI in a single component.
- **Inline styles throughout the game board** — hardcoded hex colors and `style={{}}` objects violate the project's styling rules.
- **Rules-compliance bugs** — life-out defeat triggered incorrectly, blocker validation gap, life area ordering inverted vs rules.
- **Data model gaps** — `CardData` is missing `attribute` and `types` (tribe) fields that M4 effect targeting requires.
- **Missing utility functions** — M4's effect engine needs `returnDonToDeck()`, an event-emitting `drawCards()`, and field/area predicates that don't exist yet.

This phase ships before M4 so that the effect engine plugs into clean, tested, well-separated, rules-correct code.

## Task 6: Fix Life-Out Defeat Bug — ✅ Complete

**Problem:** `battle.ts` set `damagedPlayerIndex` unconditionally whenever a leader was attacked, regardless of life count. After the damage loop removed life cards, `checkDefeat()` saw both `damagedPlayerIndex` set AND `life.length === 0` on the post-execution state — triggering a false defeat when a player went from 1+ life to 0 life. Per rule §7-1-4-1-1-1, defeat only occurs when the leader takes damage while **already** at 0 life.

**Fix:** Restructured the damage loop so the life-0 check runs at the start of each damage instance. `damagedPlayerIndex` is only set when life is already 0 at that check point. Correctly handles:
- Normal damage (1 life → 0 life = survive)
- Double Attack with 1 life (first damage removes life card, second hits 0 life = defeat)
- Already at 0 life (immediate defeat)

**Tests added** (48 total, was 46):
- "reaching 0 life from damage does NOT end the game (§7-1-4-1-1-1)"
- "Double Attack with 1 life: first damage reduces to 0, second triggers defeat"

**Files changed:** `battle.ts`

**Rules:** §1-2-1-1-1, §7-1-4-1-1-1, §7-1-4-1-1-3

## Task 7: Fix Blocker Once-Per-Battle Validation — Complete

**Problem:** `BattleContext.blockerActivated` is set to `true` when a blocker is declared, but `validateDeclareBlocker()` in `validation.ts` never checks it. A second `DECLARE_BLOCKER` in the same battle would incorrectly succeed. Per rule §7-1-2-1, the defending player can only activate [Blocker] once per battle.

**Fix:** Add a check in `validateDeclareBlocker()`: if `state.turn.battle?.blockerActivated` is `true`, reject the action.

**Test to add:** "rejects second blocker declaration in the same battle"

**Files:** `validation.ts`

**Rules:** §7-1-2-1

## Task 8: Fix Life Area Ordering — Complete

**Problem:** Rule §5-2-1-7 says life cards are placed so that "the card at the top of the deck is at the bottom in the Life area." In `setup.ts`, cards are drawn from the top of the deck and mapped directly to `life[0]`, `life[1]`, etc. But `removeTopLifeCard()` removes `life[0]` first (treating it as "top" of the life area). This means the first card drawn from the deck — which should be at the **bottom** of the life area — is instead the first card removed on damage. The ordering is inverted.

**Fix:** Reverse the life card array after drawing from deck in `setup.ts`, so that the card originally at the top of the deck ends up at the last index (bottom of life area) and is the last to be removed.

**Verification:** Add a unit test confirming the card originally at the top of the deck is the last life card removed.

**Files:** `setup.ts`

**Rules:** §2-9-2-1, §5-2-1-7, §3-10-2

## Task 10: Add M4 Foundation Utilities — Complete

**Problem:** M4's effect engine will need several utility functions that don't exist. Building them now with tests means M4 can focus on the effect system without stopping to lay plumbing.

**Functions to add:**

1. **`returnDonToDeck(state, playerIndex, count)`** — Select DON!! cards from Leader area, Character area, and cost area, and return them to the DON!! deck. Required for "DON!! −X" activation costs (§8-3-1-6, §10-2-10). No equivalent exists; `returnAttachedDonToCostArea()` returns to cost area, not DON!! deck.

2. **`drawCards(state, playerIndex, count)`** — Draw N cards from deck to hand, emitting a `CARD_DRAWN` event per card. Required for effect-driven draws like "draw 2 cards" (§4-5-3). The existing `drawN()` in `setup.ts` is setup-only and doesn't emit events or update zone metadata.

3. **`isOnField(zone)`** — Returns `true` for LEADER, CHARACTER, STAGE, COST_AREA zones. "The field" is defined in §3-1-2. Used by effect targeting ("a card on the field").

4. **`isOpenArea(zone)`** / **`isSecretArea(zone)`** — Returns whether a zone is open or secret per §3-1-5. Critical for §8-4-5: auto effects only activate when a card moves to an open area.

**Files:** `state.ts` (new functions), unit tests in `unit.test.ts`

**Rules:** §3-1-2, §3-1-5, §4-5-3, §8-3-1-6, §8-4-5, §10-2-10
