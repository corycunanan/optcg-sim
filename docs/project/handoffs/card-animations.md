#### Round 2 VQA (2026-04-21) ┬Ę PR #101
Follow-up polish on the merged primitive, still under OPT-266 scope. Public API unchanged ŌĆö consumers get these for free once they migrate.
- **DON label in rest state.** The pill was inside the card's rotating layer, so its `bottom-1 right-1` anchor rotated with the card and ended up overflowing the visible bottom-*left* of a rested card. Moved it out of the rotating layer; `right`/`bottom` insets are now computed from the card's un-rotated `(W, H)` so it tracks the *visible* bottom-right in both orientations (`card.tsx` lines ~137-155). Count badge is untouched (stacked-zone cards don't rest).
- **Motion structure split.** The rotating `motion.div` now wraps a nested interaction `motion.div`. Outer owns state rotate/opacity/filter (`cardRest`/`cardActivate`), inner owns `whileHover`/`whileTap`. Done so interaction transforms *compose* with state rotation instead of overwriting it ŌĆö rested cards wiggle around 90┬░ now, not snap back to 0┬░.
- **Hover feel.** Scale springs *in* (stiffness 420 / damping 13, faster + more pronounced than `cardActivate`) and tweens *out* (150ms easeOut) so the bounce plays on pointer-enter only. Added a single-cycle ┬▒1.2┬░ / 550ms rotate wiggle on hover-in via keyframes in `board.card.hover` / `handHover` ŌĆö cards feel alive when picked up.
- **Preview addition.** `/preview/card` now has a **Board simulation ŌĆö active Ōåö rest toggle** section: single field card with DON, inside a 112├Ś112 `SQUARE` slot matching the real character row, with a `secondary` button to flip state. Useful for VQA'ing the rotation + DON + hover combo in isolation.

  - **Sortable item IDs must match the drag payload.** Hand cards use `id: hand-${instanceId}` and `data.type: "hand-card"`, matching the existing `HandCardDrag` contract. Don't drop the `hand-` prefix ŌĆö `useBoardDnd` doesn't currently key off it, but it keeps the sortable namespace distinct from other potential drag sources.
  - **`useSortable` makes the card a droppable too.** Before this ticket, hand cards were pure draggables. Now they're sortable (draggable + droppable). Other drag types (active-don, attacker, redistribute-don) may land on hand cards; `useBoardDnd.handleDragEnd` silently ignores those combinations since it has no branch for them. If you add a new drag type that could accidentally "drop on a hand card," verify the drag-end switch has no fall-through case.
  - **Nested `DndContext` in the arrange modal is intentional.** The modal's drag interactions are self-contained and shouldn't participate in the board's context (no field zones are valid drop targets from inside the modal). If you rewire any *other* modal onto sortable, repeat the pattern ŌĆö don't reach out to the outer context. Each nested context needs its own sensors, its own `DragOverlay`, and its own `useDragTilt` instance (they use refs, so a single hook can't serve two DndContexts).
  - **`HandLayer` branches on `faceDown`.** Opponent's face-down hand skips `SortableContext` entirely (no reorder interaction, no sortable state machine cost). Player's face-up hand always wraps in SortableContext ŌĆö even when `enableDrag=false` (e.g., during opponent turn), sortable's `disabled` flag on each item just prevents drags from starting. Don't collapse the branch.
  - **`computeReorderedCustomOrder` prunes stale ids from `prev` on every reorder.** So `customOrder` can't grow unbounded with IDs the player has ever seen. The merge path in `mergeHandOrder` tolerates stale ids too (it skips any id that's not in the current server hand), but pruning at reorder time is the idempotent home for the cleanup.
  - **Reduced-motion gating.** In `SortableHandCard` + `SortableModalCard`, we force `transition: "none"` when `prefers-reduced-motion` is set so cards snap to new positions without a tween. This is the sortable transition string from dnd-kit ŌĆö overriding it is the only lever, don't try to gate at the container level.
  - **`touch-action: none`** is set on sortable cards in both surfaces (hand + modal). Needed so pointer gestures on touch don't scroll the hand/modal while dragging. Don't remove.
