# M2.5 — Design System

> Establish a consistent, tight design system grounded in the One Piece brand. Fix spacing, typography, color, and styling discipline before building the social layer and game board.

---

## Why This Phase Exists

The M1 deck builder revealed systemic design debt:

- **100+ mixed styling instances** — components use both inline `style={{}}` and Tailwind classes simultaneously, making global changes fragile
- **15+ hardcoded oklch() values** — color decisions scattered across components instead of living in tokens
- **8+ custom font sizes** (`text-[9px]`, `text-[10px]`, `text-[11px]`) — no real typographic scale
- **20+ spacing values off the 8px grid** — random `p-2.5`, `px-3 py-1.5`, `pt-4`
- **6 different border-radius values** without clear hierarchy
- **Decorative effects** (backdrop blur on badges, lift-on-hover stacking) that feel gamified rather than premium
- **Inline style manipulation via JS** (`onMouseOver` setting `borderLeftColor`) instead of CSS state

The current aesthetic direction (dark, moody, teal-tinted) doesn't match what users actually expect from a One Piece tool. The official OPTCG website and the franchise itself are bright, energetic, and warm — not somber. We're building for One Piece fans, not for a generic esports platform.

This phase ships before M2 (Social) so that every subsequent surface is built on a solid foundation.

---

## Design Direction

### References

- **Primary:** [Official OPTCG Website](https://en.onepiece-cardgame.com/) — bright white/off-white surfaces, deep navy navigation, gold accents, generous whitespace, card art as hero
- **Secondary:** [MTG Arena](https://magic.wizards.com/en/mtgarena) — sophisticated spacing, fluid typography, purposeful motion, CSS variable discipline

### Brand Personality (Revised)

**Adventurous. Warm. Confident.**

One Piece is about joy, camaraderie, and the pursuit of dreams — not grim darkness. The OPTCG Simulator should channel that energy. Card art is spectacular and deserves a clean, bright stage. The UI should feel like the official OPTCG site: professional, alive, and unmistakably One Piece.

Emotional goals: delight when browsing cards, focus when building decks, immersion during gameplay.

### Aesthetic Direction (Revised)

- **Visual tone:** Clean and bright, with purposeful dark moments (game board, modals). Inspired by the official OPTCG website — professional, welcoming, high-fidelity card presentation.
- **Primary theme:** Light mode. Dark mode surfaces reserved for the game board where immersion matters most.
- **Color palette:**
  - **Base:** Warm white (`#FAFAF8`) and off-white (`#F5F4F0`) backgrounds — not pure white
  - **Navy:** Deep navy (`oklch(22% 0.04 245)`) for navigation, headers, primary text — the franchise's structural color
  - **Gold:** Warm amber/gold (`oklch(72% 0.14 75)`) for primary accents, highlights, CTAs — treasure, adventure
  - **Red:** One Piece red (`oklch(55% 0.20 25)`) for emphasis, energy, destructive actions
  - **Card colors:** The six TCG colors (Red, Blue, Green, Purple, Black, Yellow) remain functional, not decorative
- **Typography:**
  - **Display/headings:** Barlow Condensed (Google Fonts) — bold condensed for page titles and section headers. Weights 600–800.
  - **Body:** Geist Sans — all UI text, labels, body copy.
  - Scale: 12/14/16/18/20/24/30px only. No custom `text-[Xpx]` in components.
- **Accessibility:** WCAG AA — 4.5:1 contrast for text, keyboard navigable, focus visible.
- **Card presentation:** Cards are the hero. Clean warm-white surfaces let art breathe. Hover states are subtle — no stacking of lift + shadow + blur simultaneously.
- **Anti-references:** NOT dark/moody/gamer. NOT generic SaaS dashboard. NOT neon-on-black. NOT Moxfield's utilitarian grid. The OPTCG official site's warmth and the MTG Arena site's polish — that's the target.

### Design Principles (Revised)

1. **Card art is the hero** — Clean backgrounds amplify artwork. UI chrome is minimal and recedes.
2. **One Piece warmth** — Bright, energetic, inviting. Warmth comes from color and whitespace, not decoration.
3. **Tight system, loose expression** — Every spacing, type, and color decision traces back to a token. Within that system, components can be expressive.
4. **Motion earns its place** — Transitions communicate state, not personality. One clear animation per interaction.
5. **Progressive clarity** — Simple at rest, detailed on interaction. Dense information (stats, filters) is scannable, not overwhelming.

## Scope

### Deliverables

**1. Design Token System (globals.css)**
- [ ] Revised color palette: warm white base, navy, gold, red, semantic (success/warning/error)
- [ ] Strict spacing scale: 4px increments (4, 8, 12, 16, 20, 24, 32, 40, 48, 64, 80, 96)
- [ ] Border-radius system: 3 values only — `sm` (4px), `md` (8px), `lg` (12px), `full` (9999px)
- [ ] Typography scale: 6 sizes only — `xs` (12px), `sm` (14px), `base` (16px), `lg` (18px), `xl` (20px), `2xl` (24px), `3xl` (30px)
- [ ] Shadow system: 3 levels — `sm`, `md`, `lg` — defined as tokens, not hardcoded
- [ ] Overlay token: `--overlay` for modal backdrops
- [ ] Semantic alpha variants: `--error-soft`, `--warning-soft`, `--success-soft` as tokens
- [ ] Tailwind v4 `@theme` expansion: expose all tokens as Tailwind utilities (eliminate inline style objects)

**2. Styling Discipline Rules**
- [ ] No inline `style={{}}` for design tokens — all colors via Tailwind utilities or `className`
- [ ] No hardcoded oklch/hex values in component files — use tokens
- [ ] No custom `text-[Xpx]` sizes — use scale only
- [ ] No spacing values off the scale (p-2.5, px-3 py-1.5 patterns eliminated)
- [ ] No JS-based style manipulation (onMouseOver setting inline styles)
- [ ] `cn()` utility (clsx + tailwind-merge) used for all conditional classes

**3. Base Component Library (`src/components/ui/`)**

### Spacing

Only these values. Use `gap-`, `p-`, `m-` with Tailwind's default 4px unit scale:

| Token | px | Tailwind |
|-------|----|----------|
| 1     | 4  | `p-1`    |
| 2     | 8  | `p-2`    |
| 3     | 12 | `p-3`    |
| 4     | 16 | `p-4`    |
| 5     | 20 | `p-5`    |
| 6     | 24 | `p-6`    |
| 8     | 32 | `p-8`    |
| 10    | 40 | `p-10`   |
| 12    | 48 | `p-12`   |
| 16    | 64 | `p-16`   |

No `p-2.5`, `px-3 py-1.5`, `p-3.5`, `pt-0.5`, etc. If the value isn't in this table, it's wrong.

### Border Radius

| Name  | Value  | When to use |
|-------|--------|-------------|
| `sm`  | 4px    | Badges, inline tags, small inputs |
| `md`  | 8px    | Buttons, input fields, small cards |
| `lg`  | 12px   | Panels, large cards, modals |
| `full`| 9999px | Pills, avatars, circular elements |

Never use `rounded` (4px, same as sm), `rounded-xl` (12px, same as lg by our definition), `rounded-2xl`, `rounded-3xl`.

### Typography

| Token      | Size | Weight  | Usage |
|------------|------|---------|-------|
| `label-xs` | 11px | medium  | NOT USED — use `xs` |
| `xs`       | 12px | medium  | Labels, captions, badges |
| `sm`       | 14px | regular | Secondary body, form labels |
| `base`     | 16px | regular | Primary body text |
| `lg`       | 18px | medium  | Subheadings, emphasis |
| `xl`       | 20px | semibold| Section titles |
| `2xl`      | 24px | semibold| Page headings |
| `3xl`      | 30px | bold    | Display / hero |

No `text-[9px]`, `text-[10px]`, `text-[11px]`. Minimum rendered text size is 12px (`text-xs`).
