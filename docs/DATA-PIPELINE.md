# Card Data Pipeline — vegapull & punk-records Evaluation

> Research findings on OPTCG card data sourcing, evaluation of community tools, and pipeline design decisions.

---

## Background

The OPTCG Simulator requires a complete card database covering all released sets, including card metadata, effect text, images, and art variants. Three approaches were considered:

1. **Custom scraper** — build our own scraper for the official OPTCG website
2. **Raw HTML snapshots** — manually save page source per set, parse locally
3. **Community tools** — use existing open-source scrapers and prefetched datasets

Approach 1 was attempted and abandoned due to cookie/bot protection on the official site. Approach 2 was considered as a fallback. Approach 3 was evaluated and selected, with caveats documented below.

---

## vegapull — The Scraper

**Repo:** https://github.com/coko7/vegapull
**Language:** Rust
**License:** GPL-3.0
**Install:** `cargo install vegapull`
**Last updated:** January 20, 2026 (active development)
**Stars:** 25 | Forks: 7

### What It Does

vegapull is a CLI tool that scrapes the official onepiece-cardgame.com website and outputs structured JSON. It handles the cookie/authentication issues that blocked our custom scraper attempt.

**Key commands:**
- `vega pull all` — interactive mode, downloads all packs and cards
- `vega pull packs` — downloads pack list only
- `vega pull cards {pack_id}` — downloads cards for a specific pack
- `vega pull cards {pack_id} --with-images` — includes image downloads

**Capabilities:**
- Parallel downloads (built into v1.0.0)
- 7 languages: English, English-Asia, Japanese, Chinese (HK/TW), Thai, French
- Configurable user agent
- Per-pack or all-at-once download
- Image download support
- Block number field support (added January 2026)

### Output Format (Per Card)

```json
{
  "id": "ST01-001",
  "pack_id": "569001",
  "name": "Monkey.D.Luffy",
  "rarity": "Leader",
  "category": "Leader",
  "img_url": "../images/cardlist/card/ST01-001.png?251017",
  "img_full_url": "https://en.onepiece-cardgame.com/images/cardlist/card/ST01-001.png?251017",
  "colors": ["Red"],
  "cost": 5,
  "attributes": ["Strike"],
  "power": 5000,
  "counter": null,
  "types": ["Supernovas", "Straw Hat Crew"],
  "effect": "[Activate: Main] [Once Per Turn] Give this Leader or 1 of your Characters up to 1 rested DON!! card.",
  "trigger": null
}
```

### Assessment

**Strengths:**
- Actively maintained, recently updated
- Already solved the cookie/auth problem we hit
- Covers all languages
- Clean JSON output with all card fields we need
- Rust binary — fast, no runtime dependencies

**Risks:**
- We haven't confirmed it works against the current site (cookie protections may have changed since Jan 2026)
- Depends on the official site's HTML structure — site redesign would break it
- GPL-3.0 license applies to the scraper source, not to the data it produces

## punk-records — Prefetched Dataset

**Repo:** https://github.com/buhbbl/punk-records
**License:** AGPL-3.0 (source); card data copyrighted by Bandai Namco
**Last generated:** March 8, 2026 (English)

### What It Contains

A static, versioned JSON dataset generated by vegapull. Pre-scraped card data ready for import.

**English dataset stats:**
- **50 packs** (starter decks, booster sets, promo packs)
- **4,007 unique cards** (per cards_by_id index)
- Sets covered: ST-01 through ST-27, OP-01 through OP-09, plus ultra/promo decks
- Per-card JSON files organized by pack
- Pre-built indexes: `cards_by_id.json`, `by_name.json`
- Manifest with generation timestamp

**File structure:**
```
english/
  packs.json                    # Pack metadata (id, title, label)
  cards/
    569001/                     # Pack ID directory
      ST01-001.json             # Individual card files
      ST01-002.json
      ...
    569101/
      OP01-001.json
      ...
  index/
    cards_by_id.json            # {card_id → summary}
    by_name.json                # {lowercase_name → [card_ids]}
  manifest.json                 # Generation metadata
```

### Known Limitations

1. **Incomplete set coverage:** Contains ST-01 through OP-09, but missing OP-10 through OP-14 and recent EB (Extra Booster) sets. This is a snapshot timing issue — the dataset was generated March 8, 2026 and hasn't been updated since.

2. **No art variant grouping:** Each card printing is a separate entry. A base card and its parallel art appear as separate JSON files, potentially in different pack directories. No deduplication or variant linking.

3. **No ban/restricted status:** All cards are treated equally — no format legality information.

4. **No block rotation data:** No information about which blocks/formats a card is legal in.

5. **No errata tracking:** Effect text reflects whatever was on the site at scrape time. Historical errata not captured.

6. **Effect text contains HTML:** `<br>` tags in effect text need sanitization.

7. **English dataset notes as reliable; Japanese has known issues** (wrong formatting, duplicate cards per maintainer's README).

## Art Variant Problem

### The Issue

The official site (and therefore vegapull) lists each card printing as a separate entry. A card like OP01-001 might appear in:
- OP-01 booster (base art)
- A premium/collector pack (parallel art)
- A promo pack (alternate art)

Each appearance has the same card ID but different image URLs and potentially different pack IDs. Our data model needs these grouped: one canonical `Card` record with multiple `ArtVariant` children.

### Solution: Deduplication/Grouping Transform

The import pipeline includes a grouping step:

1. **Group by card ID** — all entries with the same `id` field are the same card
2. **First occurrence = base card** — the entry from the lowest-numbered set is the canonical record (determines base stats, effect text, image)
3. **Subsequent occurrences = art variants** — additional entries become `ArtVariant` records with:
   - `variantId`: derived from pack_id + card_id
   - `label`: inferred from rarity (SecretRare → "SEC", Special → "SP", TreasureRare → "Treasure") or pack type (promo pack → "Promo")
   - `imageUrl`: the variant's image URL
   - `set`: the pack/set the variant was released in
4. **Conflict resolution** — if effect text differs between printings (rare, usually errata), flag for manual review

### Signals Available for Variant Classification

| Signal | How It Helps |
|---|---|
| `rarity` field | SecretRare, Special, TreasureRare, Promo indicate non-base printings |
| `pack_id` | Different pack than the card's "home" set = variant |
| `img_url` | Different image URL = different art |
| Card ID prefix vs pack label | e.g. OP01-001 in a pack labeled "ST-XX" = variant/reprint |

This is a transform step (~50–100 lines of code), not a scraper problem. The data is there — it just needs grouping logic.

## Pipeline Design (Revised)

### Architecture

```
                    ┌──────────────────┐
                    │   vegapull CLI    │
                    │   (Rust binary)   │
                    └────────┬─────────┘
                             │ JSON files
                    ┌────────▼─────────┐
                    │  Raw card data    │
                    │  (per-pack JSON)  │
                    └────────┬─────────┘
                             │
              ┌──────────────▼──────────────┐
              │   Import Pipeline (TypeScript) │
              │                                │
              │  1. Load JSON files             │
              │  2. Map fields to our schema    │
              │  3. Sanitize effect text        │
              │  4. Group art variants           │
              │  5. Upsert to PostgreSQL         │
              │  6. Download images → R2          │
              └──────────────┬──────────────────┘
                             │
              ┌──────────────▼──────────────┐
              │   PostgreSQL (Supabase)       │
              │   cards + art_variants tables │
              └──────────────────────────────┘
```

### Two Data Source Modes

**Mode A: Run vegapull directly (preferred)**
- Install: `cargo install vegapull`
- Run: `vega pull all` → outputs JSON to local directory
- Import pipeline reads from local directory
- Gives us ALL sets currently on the official site (including OP-10 through OP-14)
- Used for: initial setup, new set releases

**Mode B: Use punk-records snapshot (fallback)**
- Clone: `git clone https://github.com/buhbbl/punk-records.git`
- Import pipeline reads from cloned directory
- Coverage limited to what punk-records has (currently through OP-09)
- Used for: if vegapull hits cookie issues, or for testing/development

### Pipeline Steps (Detailed)

```
1. LOAD
   - Read packs.json → build pack_id-to-set-label mapping
   - Read all per-card JSON files across all packs
   - Output: array of raw card objects

2. TRANSFORM
   - Map fields per the mapping table above
   - Parse pack_id → set label (e.g. 569001 → "ST-01")
   - Map category → CardType enum
   - Filter out DON!! cards (category: "Don") — not stored in card DB

3. SANITIZE
   - Replace <br> tags with newlines in effectText
   - Strip any remaining HTML tags
   - Normalize whitespace

4. DEDUPLICATE & GROUP VARIANTS
   - Group all entries by card ID
   - For each group:
     - First occurrence (lowest set) = canonical Card
     - Additional occurrences = ArtVariant records
     - If effect text differs → flag for manual review (possible errata)

5. WRITE
   - Upsert Card records to PostgreSQL via Prisma
   - Create ArtVariant records linked to parent Card
   - Log: cards created, updated, variants created, flagged for review

6. DOWNLOAD IMAGES
   - For each Card + ArtVariant, download image from img_full_url
   - Upload to Cloudflare R2
   - Update imageUrl/altImageUrls to point to CDN
   - Rate-limit downloads to avoid overwhelming the source

7. VERIFY
   - Count total cards in DB vs source
   - Spot-check known cards (Luffy, Ace, etc.) for correct data
   - Report any cards with missing required fields
```

## Impact on Project Phases

### M0 — Foundation

**Before:** 2–3 days estimated for data pipeline (scraper + parser)
**After:** ~1 day for import pipeline (JSON transform + DB write)

The pipeline is now TypeScript (same language as the app), eliminating the Python dependency. No scraping logic, no HTML parsing, no cookie handling.

**First task in M0:** Try running `cargo install vegapull && vega pull all` to confirm it works against the current site. If it does, we have full coverage. If not, fall back to punk-records + manual supplementation.

### M1 — Deck Builder

No impact. The deck builder queries cards from PostgreSQL regardless of how they got there.

### M4 — Effect Engine

The `effect` field in punk-records contains the full English card text — exactly what we need for hand-authoring effectSchema and for LLM-assisted parsing in M5. The text includes keyword markers like `[Rush]`, `[Blocker]`, `[DON!! x2]`, `[When Attacking]` etc., which makes parsing more tractable.

Example effect text patterns observed in the data:
- `"[Rush] (This card can attack on the turn in which it is played.)"`
- `"[DON!! x2] [When Attacking] Your opponent cannot activate [Blocker] during this battle."`
- `"[Activate: Main] [Once Per Turn] Give this Leader or 1 of your Characters up to 1 rested DON!! card."`
- `"[Main] K.O. up to 1 of your opponent's Characters with 6000 power or less."`
- `"[Counter] Up to 1 of your Leader or Character cards gains +2000 power during this battle."`

The bracketed keywords in the raw text are machine-parseable and directly correspond to our effect trigger types. This is a good signal for LLM parsing accuracy.

### M5 — Polish & Scale

**Before:** LLM parser needed to handle 1,500+ cards
**After:** Same scope, but the input data is cleaner (structured JSON with explicit fields rather than raw HTML). The `effect` field gives the LLM exactly the text it needs without extraction noise.

For new set releases: run `vega pull cards {pack_id}` for the new pack, pipe through the import pipeline. Target: under 30 minutes per new set.

## Open Questions

- [ ] **Does vegapull work today?** Need to test `vega pull all` against the current site. Cookie protections may have changed since the tool's last update (Jan 2026).
- [ ] **Image hosting rights:** Can we download and re-host card images on our CDN? Need to understand the copyright implications. The images are copyrighted by Bandai Namco. Other fan projects (sim tools, deck builders) generally do this — but it's worth noting.
- [ ] **Errata tracking source:** Where do we get errata data? The official site may update card text in-place without history. We may need to diff against our stored data on each re-import to detect changes.
- [ ] **Block rotation data source:** Is there an official list of which cards are legal in which block formats? This may need to be maintained from official tournament announcements.
